# PROJECT_NORTH_STAR.md

> 这是《Slay the Spire 2》AI 智能体项目的长期纲领。
> 它定义项目的核心思想、产品方向、智能体哲学、认知脚手架、工程边界、学习准则和长期成功标准。
> 它应该很少变动。
> 它不记录临时实现细节、当前运行状态、具体命令输出、短期里程碑、调试偶发问题或某一轮开发计划。
> 操作计划、数据模式、命令、适配器、实现状态、实验结果、调试记录和短期任务，应该放在项目计划、架构文档、schema 文档、handoff 文档和工程笔记中。

## 1. 北极星

本项目的目标，是构建一个真实游戏中的《Slay the Spire 2》智能体：**LLM 始终是核心战略玩家，而本地系统作为预测型认知脚手架，最大化 LLM 能够感知、记忆、想象、判断、执行、复盘和学习的能力。**

本项目不是追求“任意算法意义上的最强 AI”。

它追求的是：

```text
在 LLM 作为核心战略玩家的前提下，
通过外部认知脚手架，
让 LLM 逐步接近它自身能够达到的最强战略水平。
```

这个项目不应该退化成规则机器人。
它不应该把真正的玩家隐藏在手写策略引擎后面。
它不应该让黑箱 solver 悄悄替代 LLM 做所有重要选择。
它也不应该把原始游戏状态直接倾倒给 LLM，然后期待智能自然出现。

真正的目标是：

```text
让 LLM 看见强玩家会看见的局面，
想象强玩家会想象的未来，
记住真正会改变未来判断的经验，
在经过验证的候选未来之间进行战略深思，
并通过预测误差不断改进它下一次看见、想象、判断和行动的方式。
```

项目的核心闭环是：

```text
真实游戏环境
  -> 规范状态
  -> 战略第一印象 / 显著性压缩
  -> 记忆激活
  -> 候选未来预测
  -> LLM 战略深思
  -> 经过验证的安全执行
  -> 转换记录
  -> 回放 / 评估 / 复盘
  -> 预测误差归因
  -> 记忆、候选生成、上下文压缩和策略判断的受控改进
```

长期核心问题是：

```text
一个 LLM 能否从很少的游戏专属经验开始，
通过结构化脚手架游玩真实游戏，
观察结果，
把经验转化为记忆、派生知识、候选模板和战略抽象，
改进自己第一眼看局面、激活经验、想象未来和评估风险收益的方式，
并逐步接近它自身能够达到的最强玩法水平？
```

长期成功标准是：

* **LLM 潜能释放：** 脚手架增强 LLM 的有效战略能力，而不是让 LLM 变得无关紧要。
* **战略第一印象改进：** LLM 看到的不再是原始状态，而是越来越接近强玩家会注意到的战略态势。
* **候选未来改进：** 候选生成不只是列出合法动作，而是越来越能预测计划的收益、代价、风险和失效条件。
* **记忆质量改进：** 记忆不只是日志，而是有条件、有证据、有置信度、能影响未来判断的经验。
* **战略质量提升：** 智能体在战斗、奖励、路线、商店、事件、休息点、药水、资源和长期牌组发展中做出越来越强的决策。
* **学习闭环成立：** 重复游玩会改进上下文压缩、记忆检索、候选生成、预测质量、LLM 深思和未来决策。
* **可靠执行：** 每个动作都合法、基于当前状态，并且在有意义的不确定性后能够重新读取和恢复。
* **可观测和可复盘：** 决策被记录为可回放、可评估、可归因的状态 / 预测 / 动作 / 结果转换。
* **可维护：** 项目随着成长应该更容易理解，而不是更像一堆难以审计的 prompt、规则和日志。

## 2. 项目身份

这个项目不是一个 prompt。
不是单个 controller。
不是一个 mod。
不是一个硬编码策略脚本。
不是一组启发式规则。
不是一个让 LLM 背锅的自动化脚本。
也不是一个把 LLM 包在外壳里的传统 solver。

它是一个 **LLM-centered predictive cognitive scaffold**：

```text
以 LLM 为中心的预测型认知脚手架系统。
```

它的核心不是“本地系统替 LLM 玩游戏”，而是：

```text
本地系统塑造 LLM 面对问题的方式。
```

本地系统应该帮助 LLM 完成四件事：

```text
看见：
  从原始状态中形成战略第一印象。

记起：
  激活与当前局面真正相关的经验、教训、条件和模式。

想象：
  生成若干经过验证的候选未来，而不只是列出当前可点的按钮。

深思：
  让 LLM 在紧凑但战略完整的上下文中，进行理性权衡和选择。
```

产品的持久心智模型是：

```text
Game integration
  -> canonical state
  -> mechanics and facts
  -> strategic impression
  -> memory activation
  -> candidate futures
  -> deliberation packet
  -> LLM-centered decision
  -> validated execution
  -> transition recording
  -> replay / eval / review
  -> prediction-error-driven learning
```

每一个未来功能都应该强化这个循环。

任何绕过这个循环、遮蔽这个循环、悄悄替代 LLM 战略判断、破坏数据真相，或者让失败更难归因的功能，都应该被拒绝、隔离，或者明确标记为实验性 baseline。

## 3. 核心认知启发

本项目可以借鉴认知科学、神经科学、强化学习和 LLM agent 研究中的一些思想，但这些思想只作为工程启发，不作为生物学声明。

本项目不声称：

* LLM 是人类大脑；
* prompt 是意识；
* 向量数据库是海马体；
* 候选生成器就是人类想象力；
* context compression 就是生物感知；
* agent 复刻了人类认知。

这些类比的价值在于，它们帮助项目提出正确的工程问题。

本项目真正关心的是：

```text
智能体第一眼应该注意到什么？
哪些经验应该被当前局面激活？
当前局面可能通向哪些未来？
这些未来的收益、代价、风险和不确定性是什么？
LLM 应该看到什么，才能做出强玩家式判断？
系统之前预测了什么？
实际发生了什么？
预测误差应该归因给哪个模块？
下一次应该如何看得更准、想得更全、记得更对、选得更好？
```

因此，所有认知隐喻最终都必须落成：

* 可检查的软件对象；
* 可记录的数据结构；
* 可复盘的 transition；
* 可测试的行为；
* 可回滚的更新；
* 可评估的改进。

如果一个隐喻不能帮助实现、诊断或评估，它应该留在背景哲学中，而不应该成为架构负担。

## 4. 第一印象原则：Context Compression 是战略感知

Context compression 不是普通摘要。

它的目的不是简单减少 token。
它的目的不是把状态剪短。
它的目的不是隐藏复杂性。

它的目的是把原始游戏状态转化成 LLM 可以用于战略思考的第一印象。

弱压缩器会说：

```text
给 LLM 更少的信息。
```

强压缩器会说：

```text
给 LLM 正确的抽象。
```

在本项目中，context compression 应该产生 **Strategic Impression**：

```text
战略第一印象。
```

战略第一印象应该回答：

```text
这个局面第一眼是什么？
这是普通决策，还是关键决策？
哪里危险？
哪里有机会？
哪里不确定？
当前牌组的问题是什么？
当前路线的问题是什么？
当前资源的问题是什么？
这个局面像过去哪次成功或失败？
LLM 是否需要深思？
```

例如，原始状态可能是：

```text
敌人攻击 18。
玩家血量 42。
手牌 A/B/C/D。
有一瓶药水。
下一层可能进精英。
```

战略第一印象不应该只是复述这些字段，而应该变成：

```text
这是一个潜在资源权衡回合。
本回合可以低损防住，但使用药水可能直接结束战斗。
药水对下一层精英可能更重要。
当前牌组防御密度偏低，这不是单回合问题，而是结构性风险。
过去类似低防御密度局面中，过度贪伤害导致后续精英前血量崩盘。
```

这才是 LLM 需要看到的“局面”。

一个遗漏决定性特征的短 prompt，比一个稍长但保留真实战略问题的 prompt 更差。

## 5. 显著性原则：不是所有信息都值得进入意识

智能体不应该把所有状态字段视为同等重要。

强玩家不是看见更多原始字段，而是更快看见真正重要的东西。

因此，系统需要显著性判断。

显著性可以来自：

```text
危险：
  当前死亡风险、下回合死亡风险、低血量、不可避免伤害。

价值波动：
  某个选择可能显著改变未来胜率，例如关键 relic、关键删牌、关键 scaling。

新颖性：
  当前局面与常见局面不同，例如异常遗物组合、奇怪牌组方向、特殊事件。

不确定性：
  候选项排序不清楚，局部估算冲突，未来风险不明确。

记忆共鸣：
  当前局面触发过去类似成功、失败、错误或教训。

不可逆性：
  路线锁定、错过商店、跳过关键牌、失去药水、错过删除等难以弥补的选择。

重复失败模式：
  系统曾多次在类似局面犯错。
```

显著性不是生物学公式，而是工程信号。

它应该影响：

* 什么进入战略第一印象；
* 什么触发记忆检索；
* 什么进入 LLM 深思包；
* 哪些 transition 需要重点回放；
* 哪些错误需要写入长期学习；
* 哪些候选未来必须被生成。

---

## 6. 记忆原则：Memory 是未来想象的素材

记忆不是日志仓库。

记忆不是“把所有历史都存起来”。
记忆不是“把每次复盘总结都塞进 prompt”。
记忆也不是越多越好。

记忆的核心价值是：

```text
帮助系统更好地预测未来。
```

好的记忆应该改变下一次的第一印象、候选未来和 LLM 判断。

当系统遇到当前局面时，记忆系统应该能帮助回答：

```text
这个局面像过去哪几次？
过去类似局面后来发生了什么？
哪些早期信号预示了成功或失败？
当时缺失了什么候选项？
当时 prompt 漏掉了什么信息？
当时 LLM 判断错在哪里？
这次是否正在重复同一种错误？
```

记忆至少应该支持四种角色：

```text
事件记忆：
  具体 run、具体战斗、具体选择、具体结果、具体 transition。

语义记忆：
  抽象规律，例如某类牌组缺 AoE 时 Act 2 风险升高。

程序记忆：
  可复用决策流程，例如低血打精英前的路线评估流程。

显著性记忆：
  高危险、高价值、高惊讶、高错误、反复失败或强烈成功的关键局面。
```

记忆必须带条件。

一条经验不能只写：

```text
以后不要跳牌。
```

它应该写成：

```text
在某类牌组、某个阶段、某种 boss 压力、某种牌组厚度和某种资源状态下，
跳过某类修复牌可能增加后续风险。
证据来自哪些 run？
反例是什么？
置信度是多少？
什么时候不适用？
```

没有条件的记忆会变成迷信。

记忆应该有：

* 来源；
* 证据；
* 置信度；
* 适用条件；
* 已知反例；
* 衰减机制；
* 冲突处理；
* 回滚路径。

记忆应该影响：

* 战略第一印象；
* 候选未来生成；
* 候选排序；
* LLM 深思；
* 回放重点；
* 派生知识提案。

记忆不能污染事实层。
记忆不能伪装成客观真相。
记忆不能无条件进入 prompt。
记忆不能因为听起来合理就进入稳定策略。

## 7. 候选未来原则：Candidate Generator 是未来预测器

Candidate generator 不是合法动作枚举器。

合法动作回答：

```text
现在能做什么？
```

候选未来回答：

```text
如果选择这条计划，可能会发生什么？
```

本项目中的 candidate generation 应该从“动作列表”升级为“候选未来生成”。

候选未来应该包含：

```text
计划：
  要做什么。

预测结果：
  可能得到什么。

代价：
  会损失什么。

风险：
  什么情况下会变差。

不确定性：
  哪些估算不稳定。

假设：
  这个预测依赖什么前提。

记忆链接：
  过去哪些经验支持或警告这个未来。

失效触发：
  抽牌、生成、敌人死亡、遗物触发、目标变化、界面变化等何时会使计划失效。

执行要求：
  该计划如何转化为安全、合法、基于当前状态的动作。
```

一个成熟的候选未来不是：

```text
Candidate A: 打出 Strike。
Candidate B: 打出 Defend。
Candidate C: 结束回合。
```

而是：

```text
Candidate A: 保守生存线
  本回合少受伤，保留药水。
  代价是战斗延长，下回合仍有击杀压力。
  适合血量较低、下一层有精英、药水价值较高时。

Candidate B: 药水斩杀线
  本回合结束战斗，避免后续抽牌风险。
  代价是失去对下一场精英很有价值的药水。
  适合当前血量极低或下回合死亡风险较高时。

Candidate C: 贪伤害线
  尝试压低敌人血量，保留部分资源。
  风险是如果下回合抽不到防御或攻击，可能承担更高损失。
```

候选生成应该覆盖不同层级：

```text
Level 1: 合法动作候选
  当前可以做什么。

Level 2: 战术计划候选
  当前战斗几步内怎么打。

Level 3: 战略未来候选
  当前选择如何影响牌组、路线、资源、Boss、商店和后续风险。
```

如果候选生成很浅，LLM 的上限就会变成候选覆盖率，而不是 LLM 的战略能力。

因此，candidate generator 是学习系统的一部分。

失败后不只要问：

```text
LLM 为什么选错？
```

还要问：

```text
我们是否给了 LLM 足够好的未来可选？
有没有强候选未来根本没有被生成？
候选是否过于局部？
候选是否没有预测长期风险？
候选是否缺少记忆支持或反例？
LLM 是否请求了缺失候选？
```

经过验证的候选项是安全必需品，但它们不能变成牢笼。

## 8. 第一印象、记忆、候选未来的循环关系

Context compression、Memory、Candidate generator 不是三个独立模块。

它们应该形成循环。

```text
第一印象决定最初注意什么。
记忆决定哪些过去经验被激活。
候选生成决定这些经验被展开成哪些未来。
候选之间的分歧反过来暴露第一印象缺了什么。
预测误差再反过来修正第一印象、记忆和候选生成。
```

系统不应该只做单向流程：

```text
压缩状态 -> 检索记忆 -> 生成候选 -> LLM 选择
```

更强的流程应该是：

```text
初始战略第一印象
  -> 显著性判断
  -> 记忆激活
  -> 候选未来生成
  -> 候选分歧分析
  -> 补充战略上下文
  -> LLM 深思
```

例如，一个浅层系统可能看到：

```text
这是一个普通卡牌奖励。
```

但候选未来生成后发现：

```text
Candidate A 解决 frontload。
Candidate B 解决 scaling。
Candidate C 保持 deck consistency。
```

记忆系统又发现：

```text
过去类似 deck 多次因为没有 scaling 死于 Boss。
```

于是战略第一印象应该被更新为：

```text
这不是普通卡牌奖励。
这是 deck 方向分叉点。
当前 frontload 尚可，但 Boss scaling 不足。
短期稳定和长期成长发生冲突。
```

这就是认知脚手架的核心价值：
它不是把状态压短，而是让 LLM 看到真正的问题。

## 9. LLM 深思原则：LLM 是战略工作空间

LLM 是核心战略玩家，但它不应该独自承担所有底层感知、计算和记忆检索。

在本系统中，LLM 更像一个战略工作空间：

```text
Context compression 决定什么进入工作空间。
Memory retrieval 把相关经验带入工作空间。
Candidate generator 把可能未来带入工作空间。
Mechanics engine 把确定性计算带入工作空间。
LLM 在工作空间中进行显式权衡、冲突解决和战略判断。
Executor 把选择转化为安全动作。
Replay / eval 让结果反过来修正未来工作空间的内容。
```

LLM 的深思包应该包含：

* 战略第一印象；
* 显著风险；
* 相关记忆；
* 相关派生知识；
* 候选未来；
* 确定性计算；
* 关键权衡；
* 不确定性；
* 输出 schema；
* 验证约束。

LLM 应该能够：

* 选择一个候选未来；
* 拒绝所有候选，因为候选太浅；
* 请求缺失的候选类型；
* 指出关键不确定性；
* 说明简短战略理由；
* 建议未来脚手架如何改进。

LLM 不应该需要解析：

* 原始适配器噪声；
* 完整事实数据库；
* 全部历史；
* 全量记忆；
* 无边界复盘文本；
* 未压缩的状态 dump。

如果进入深思包的是错误问题，LLM 的回答可能逻辑连贯，但战略上错误。

---

## 10. 预测误差原则：学习来自“哪里预测错了”

项目不应该只问：

```text
赢了还是输了？
```

也不应该只问：

```text
LLM 选得对不对？
```

它应该问：

```text
我们之前预测了什么？
实际发生了什么？
预测错在哪里？
错误来自哪个模块？
下次应该如何改变？
```

预测误差可以包括：

* 伤害预测误差；
* 格挡预测误差；
* 斩杀预测误差；
* 抽牌预测误差；
* 资源预测误差；
* 战斗风险预测误差；
* 路线风险预测误差；
* 牌组质量预测误差；
* 候选未来结果预测误差；
* 记忆相关性误差；
* 上下文遗漏误差；
* LLM 判断误差；
* 执行状态误差。

预测误差必须被归因。

如果机制错了，修机制层。
如果第一印象漏掉关键特征，修 context compression。
如果强候选未来没有出现，修 candidate generator。
如果记忆误导了判断，修记忆条件、置信度或检索。
如果 LLM 在信息充分时仍然选错，写入战略教训。
如果执行依赖过期状态，修执行和 checkpoint 机制。

学习不是把失败总结写得更长。

学习是让下一次：

```text
第一眼看到的重点更准；
激活的记忆更相关；
生成的候选未来更强；
LLM 看到的权衡更完整；
预测结果更接近真实；
执行更安全；
复盘更能定位原因。
```

---

## 11. 回放原则：Replay 是离线认知训练

Replay 不是漂亮的运行总结。

Replay 是系统重新构造重要决策、检查预测误差、定位失败模块，并提出受控更新的过程。

Replay 应该重建：

```text
preState
  -> strategic impression
  -> salience signals
  -> memory activation
  -> candidate futures
  -> deliberation packet
  -> LLM decision
  -> selected action
  -> postState
  -> stateDiff
  -> prediction error
  -> error attribution
  -> proposed update
```

Replay 应该优先处理高显著性 transition：

* 死亡前几回合；
* 濒死回合；
* 大量掉血回合；
* 关键卡牌奖励；
* 错误跳牌；
* 错误删牌；
* 商店错购；
* 药水使用错误；
* 路线锁定；
* 精英前决策；
* Boss 前准备失败；
* 某个 deck archetype 崩盘的早期信号；
* 重复出现的失败模式。

Replay 应该问：

```text
当时的第一印象是否正确？
显著性判断是否抓住了重点？
是否检索到了正确记忆？
是否生成了正确候选未来？
候选未来是否预测了真实风险？
LLM 是否获得了足够信息？
LLM 是否做出了合理权衡？
执行是否保持安全？
预测误差来自哪里？
哪个模块应该改变？
```

Replay 的输出不应该只是“复盘文字”。

它应该产生受控更新：

* memory update；
* derived knowledge proposal；
* candidate template proposal；
* context feature proposal；
* mechanics test；
* replay fixture；
* evaluation case；
* prompt improvement suggestion；
* rollback note。

---

## 12. 学习原则：越玩越会不是多存文字

系统应该通过游玩、回放、评估和受控更新来改进。

学习不等于：

```text
记忆越来越多。
prompt 越来越长。
复盘越来越漂亮。
```

学习意味着：

```text
下一次遇到类似局面时，
第一印象变得更准确，
激活的记忆更相关，
候选未来更强，
LLM 看到的权衡更完整，
预测误差更小，
执行更安全，
长期决策更好。
```

学习闭环是：

```text
run experience
  -> transition records
  -> replay and review
  -> missing-impression detection
  -> missing-memory detection
  -> missing-candidate detection
  -> prediction-error attribution
  -> memory / derived knowledge / candidate template / context feature updates
  -> better future deliberation
```

项目应该支持从弱初始知识到更强玩法的启动路径。

一开始，智能体可能只有：

```text
很少游戏专属记忆；
薄弱派生知识；
简单候选生成；
基础机制计算；
有限战斗规划；
粗糙上下文压缩。
```

通过重复游玩，它应该逐步拥有：

```text
更好的战略第一印象；
更相关的记忆激活；
更丰富的候选未来；
更稳定的候选预测；
更可靠的 LLM 深思包；
更清晰的错误归因；
更安全的执行；
更有证据的长期策略。
```

学习应该分为：

```text
实验层：
  允许激进假设、异常路线、黑箱发现、探索性候选模板和统计实验。

晋升层：
  测试这些发现是否在相关情境中改善结果。

稳定层：
  只接受带有证据、置信度、条件、评估和回滚路径的更新。
```

稳定系统要保守。
实验系统不能被窒息。
两者必须隔离。

---

## 13. Reflexion、Voyager 与 ToT 的工程启发

本项目可以借鉴一些 LLM agent 方向，但不能生搬硬套。

### 13.1 Reflexion 式启发

Reflexion 的启发是：
LLM 可以不更新模型权重，而通过语言化反馈、反思和 episodic memory 改善后续决策。

对本项目而言，这意味着：

```text
失败后不一定立刻训练模型。
可以先把预测误差、错误归因和修正建议写入结构化记忆。
```

但本项目不能只做自然语言反思。
反思必须绑定到 transition、候选未来、预测误差和适用条件。

### 13.2 Voyager 式启发

Voyager 的启发是：
经验最好沉淀为可复用技能，而不是普通日志。

对本项目而言，记忆最终应该帮助形成：

```text
detect_deck_deficit(...)
generate_survival_line(...)
evaluate_card_reward(...)
evaluate_route_risk(...)
detect_missing_scaling(...)
estimate_shop_remove_value(...)
generate_boss_preparation_candidates(...)
```

也就是说，真正有价值的经验应该改变系统如何看局面、如何生成候选、如何预测未来，而不仅仅是写下一段经验总结。

### 13.3 Tree-of-Thoughts 式启发

Tree-of-Thoughts 的启发是：
复杂决策不能只靠一次性生成。
系统应该允许多个候选路径、前瞻、自评和必要时的回溯。

对本项目而言，这意味着：

```text
LLM 不应该只在 A/B/C 之间选择。
它应该能比较多个候选未来，
要求生成缺失路线，
质疑候选假设，
并在关键局面进行更深的战略搜索。
```

这些研究方向是启发，不是架构复制。

本项目的特殊性在于：
《Slay the Spire 2》是真实游戏、强约束、长程规划、高风险执行环境。
因此，所有反思、技能、候选和深思都必须受到合法性验证、状态记录、执行安全、回放评估和数据真相纪律的约束。

---

## 14. 工程落地原则：启发必须变成对象

本项目的哲学中心是预测型认知脚手架。
但哲学必须落成工程对象。

核心对象至少包括：

```text
StrategicImpression:
  当前局面的战略第一印象。

SalienceSignal:
  哪些风险、机会、不确定性或记忆共鸣值得注意。

MemoryActivation:
  当前局面激活了哪些经验，以及为什么相关。

CandidateFuture:
  一个合法计划及其预测收益、代价、风险、假设和失效条件。

DeliberationPacket:
  提供给 LLM 的紧凑战略工作上下文。

PredictionErrorRecord:
  预测和实际之间的差异，以及错误归因。

ReplayFrame:
  可重建、可复盘、可评估的重要决策帧。

ConsolidationRecord:
  本次经验应该如何改变记忆、派生知识、候选模板或上下文特征。
```

这些对象不一定一开始全部成熟。
但项目应该朝这个方向演化。

只要一个启发无法落成对象、记录、测试或评估，它就还不是工程架构。

---

## 15. 分层原则

项目必须保持清晰的依赖方向。
低层不得依赖高层。
每一层必须有清晰的职责、输入、输出和失败模式。

### 15.1 游戏集成层

负责和游戏或 mod 通信。

可以：

* 读取原始游戏状态；
* 执行动作；
* 暴露动作结果；
* 读取事件日志；
* 报告适配器能力；
* 处理连接、等待、结算和错误。

不得：

* 包含策略；
* 调用 LLM；
* 修改记忆或派生知识；
* 决定奖励、路线、商店或战斗策略；
* 把原始外部格式直接暴露给核心智能体。

外部游戏项目是适配器，不是架构。

### 15.2 规范状态与机制层

负责把不稳定外部状态变成稳定内部状态，并执行确定性推理。

可以：

* 规范化 raw state；
* 定义 canonical state / action / event / transition；
* 验证合法性、能量、目标、费用、选项；
* 计算伤害、格挡、斩杀、来袭伤害、抽牌概率和简单风险；
* 计算状态差异；
* 识别 checkpoint；
* 查询客观事实。

不得：

* 调用 LLM；
* 使用长期记忆作为机制事实；
* 把学习到的策略写进事实层；
* 用过度自信的估算隐藏不确定性。

### 15.3 第一印象与显著性层

负责把规范状态转成战略第一印象。

可以：

* 识别当前决策类型；
* 识别危险、机会、不确定性；
* 识别牌组缺陷；
* 识别资源压力；
* 识别路线压力；
* 识别记忆共鸣；
* 判断是否需要 LLM 深思。

不得：

* 为了短而漏掉决定性信息；
* 把不确定判断伪装成事实；
* 用无关经验淹没 prompt；
* 隐藏关键假设。

### 15.4 记忆与派生知识层

负责检索、组织和更新经验。

可以：

* 检索事件记忆；
* 检索语义经验；
* 检索程序模板；
* 检索显著性记忆；
* 提出派生知识；
* 降低错误记忆置信度；
* 处理冲突、衰减和适用条件。

不得：

* 把经验当作客观事实；
* 无条件泛化一次失败；
* 让低置信经验直接主导稳定策略；
* 把所有记忆塞进实时 prompt。

### 15.5 候选未来层

负责生成合法、可验证、基于状态的候选未来。

可以：

* 生成合法动作；
* 生成多步计划；
* 生成战斗路线；
* 生成奖励、路线、商店、升级、休息等战略候选；
* 预测收益、代价、风险、不确定性；
* 附加假设、记忆链接和失效触发；
* 识别缺失候选类型。

不得：

* 执行任意 LLM 编造动作；
* 绕过合法性验证；
* 混淆“推测候选”和“可执行计划”；
* 用浅层候选限制 LLM 上限而不暴露问题。

### 15.6 LLM 深思层

负责让 LLM 在正确上下文中做战略判断。

可以：

* 构造 deliberation packet；
* 调用 LLM；
* 让 LLM 选择、拒绝或请求候选；
* 让 LLM 给出简短理由；
* 让 LLM 指出不确定性；
* 让 LLM 提出脚手架改进建议。

不得：

* 让 LLM 直接操作游戏；
* 接受未验证动作；
* 依赖原始状态 dump；
* 把 LLM 输出视为自动真实或自动可执行。

### 15.7 执行与 checkpoint 层

负责安全执行。

可以：

* 执行计划动作；
* 动作后重新读取状态；
* 检测硬 checkpoint；
* 停止、继续或重新规划；
* 记录执行结果。

不得：

* 假设多动作计划在抽牌、生成、敌人死亡、目标变化、遗物触发、能量变化或界面变化后仍然有效；
* 依赖过期卡牌索引；
* 在没有重新读取状态的情况下越过硬 checkpoint；
* 把执行成功当作战略正确。

### 15.8 回放、评估与学习层

负责记录、复盘、归因和受控更新。

可以：

* 记录 transition；
* 重建 replay frame；
* 计算 prediction error；
* 归因错误模块；
* 提出 memory / derived knowledge / candidate / context 更新；
* 维护实验日志；
* 维护回滚路径。

不得：

* 把一次运行当作确定证明；
* 自动把反思写成稳定规则；
* 以无法评估或回滚的方式更新策略；
* 把人类差异推断当作精确动作真相。

---

## 16. 数据真相原则

项目必须区分：

```text
事实
观察
推断
记忆
派生知识
策略假设
LLM 反思
真实执行动作
```

这些东西不能混在一起。

### 16.1 事实

事实包括卡牌名称、费用、描述、遗物效果、关键词、角色、药水、敌人和游戏文本。

事实属于 fact layer。

学习到的信念不能写进 fact layer。

### 16.2 观察

观察是系统实际读到的状态、日志、快照和适配器输出。

观察可能不完整、过期或有噪声。

观察不等于真相。

### 16.3 推断

推断是对可能发生了什么的解释。

推断必须携带不确定性。

推断出来的人类动作、死亡原因、错失机会或策略失败，都不能直接当作真实值。

### 16.4 记忆

记忆是经验。

记忆可以影响判断，但必须有条件、证据和置信度。

### 16.5 派生知识

派生知识是“什么时候什么事实重要”。

它可以演化，但不能伪装成事实。

### 16.6 LLM 反思

LLM 反思是候选解释和改进建议。

它有价值，但不是事实。

### 16.7 执行动作

通过执行器发送的 agent 动作，可以作为所选动作的真实值。

mod / MCP 事件动作只有在身份和时间信息足够时，才能作为真实值。

---

## 17. Transition 记录原则

最终，一个重要决策应该被记录成：

```text
preState
  -> strategicImpression
  -> salienceSignals
  -> memoryActivation
  -> candidateFutures
  -> deliberationPacket
  -> llmDecision
  -> selectedPlan
  -> executedAction
  -> postState
  -> stateDiff
  -> predictionError
  -> auditMetadata
```

Transition 是回放、评估、奖励、复盘和学习的基础。

只有快照不是动作。
只有总结不是学习。
没有 preState / decision / postState，就很难真正复盘。
没有 prediction error，就很难知道该修哪个模块。
没有 audit metadata，就很难信任长期学习。

---

## 18. 战斗原则

战斗应该快速，但不能鲁莽。

长期战斗模型是：

```text
候选战斗未来
  -> 执行安全初始片段
  -> 重新读取状态
  -> 识别 checkpoint
  -> 继续、重新规划或询问 LLM
```

普通战斗不应该每张牌都问 LLM。

但系统也不能盲目执行长链计划，因为以下变化都可能让计划失效：

* 抽牌；
* 卡牌生成；
* 发现牌；
* 敌人死亡；
* 遗物触发；
* 能量变化；
* 目标变化；
* 意图变化；
* 界面变化；
* 奖励索引变化。

成熟战斗候选未来应该考虑：

* 斩杀路线；
* 存活路线；
* 伤害 / 格挡权衡；
* 敌人意图；
* 能量；
* 抽牌和弃牌状态；
* 药水；
* 遗物触发；
* 未来回合风险；
* 资源保留；
* 计划失效条件。

战斗系统的目标不是让 LLM 每一步微操。
目标是让 LLM 在真正需要战略判断时看到高质量候选未来。

---

## 19. 评估原则

项目应该评估整个认知脚手架闭环，而不只是最终胜负。

胜利很重要，但胜负太粗糙，不能独自指导学习。

评估应该检查：

* 第一印象质量；
* 显著性判断质量；
* 记忆检索相关性；
* 记忆条件准确性；
* 候选未来覆盖率；
* 候选预测质量；
* LLM 深思质量；
* 执行安全；
* checkpoint 行为；
* transition 记录质量；
* replay 质量；
* prediction error 归因质量；
* 学习更新质量；
* 重复失败是否减少；
* 随时间是否变强。

评估系统应该帮助回答：

```text
变强是因为 LLM 判断更好了？
还是因为第一印象更准？
还是因为记忆检索更相关？
还是因为候选未来更强？
还是因为机制更准确？
还是因为执行更安全？
还是因为复盘更会定位错误？
```

项目应该偏好证据，而不是感觉。

一个策略改进只有在可以被观察、回放、比较和回滚之后，才算成熟。

---

## 20. 外部依赖原则

外部项目和服务提供能力，但不定义智能体核心。

每个外部依赖都必须隔离在 adapter 后面。

项目应该知道每个外部依赖：

* 能可靠做什么；
* 不能可靠做什么；
* 什么未知；
* 有什么许可证或维护风险；
* 如何替换；
* 缺失能力时用什么 fallback。

不要因为某个能力有用，就假设外部项目一定提供。

能力必须被检测、记录和测试。

---

## 21. Mod 边界原则

有些能力属于游戏侧 mod。
大多数智能体智能不属于 mod。

mod 可以提供：

* 原始游戏状态；
* 动作执行；
* 动作结果；
* 合法动作暴露；
* 游戏事件日志；
* 人类 UI / 动作事件捕获；
* 卡牌实例 ID；
* 目标 ID；
* 选项 ID；
* 界面身份；
* 前后状态引用或 hash。

mod 不应该包含：

* LLM 调用；
* prompt；
* API key；
* 长期记忆；
* 派生策略知识；
* reward logic；
* 实验管理；
* 路线规划；
* 商店规划；
* 卡牌奖励规划；
* 战斗规划；
* 运行后复盘；
* 向量数据库；
* 训练数据导出政策。

mod 应该是传感器和执行器。
本地 agent 系统才是大脑和学习循环。

---

## 22. 人类数据原则

人类游玩数据只有在可靠性明确时才有价值。

理想长期设计是：

```text
游戏侧事件日志 mod
  捕获真实人类 / 游戏事件

本地 HumanPlayRecorder
  将事件与 preState / postState 连接，并写入 transition
```

如果没有事件日志，可以使用快照轮询和状态差异推断，但必须标记为 inferred，不能当作 ground truth。

项目绝不能悄悄把模糊的人类状态变化转换成高置信度标签。

人类数据可以帮助 LLM 和脚手架学习，但不能破坏观察、推断、动作真相和 verified transition 之间的边界。

---

## 23. 工程原则

智能体是长期软件系统，必须保持可维护。

永久工程规则：

* 保持职责分离；
* 保持外部依赖在 adapter 后面；
* 保持 schema 版本化；
* 保持 prompt 集中且经过验证；
* 保持数据格式持久；
* 保持 replay 和 evaluation 可能；
* 保持底层机制经过测试；
* 将实时策略问题和程序 bug 分开；
* 保持文档与行为一致；
* 保持补丁风险、测试和回滚明确；
* 保持系统能被未来编码智能体理解。

一个变更直到可观测，才算成熟。

一个无法回放、衡量或回滚的策略改进，不可信。

一个让系统更难推理的聪明模块，除非收益清晰且被衡量，否则应该被视为负担。

---

## 24. 绝不能被优化掉的内容

未来变更不得移除或削弱以下原则：

1. **LLM 核心战略玩家地位。** 脚手架释放 LLM 潜能，而不是让 LLM 变得无关紧要。
2. **预测型认知脚手架。** 系统核心是第一印象、记忆激活、候选未来、深思、执行、预测误差和学习闭环。
3. **战略第一印象。** Context compression 必须生成战略感知，而不是普通摘要。
4. **显著性判断。** 系统必须知道什么值得注意、什么值得进入 LLM 深思。
5. **记忆作为未来预测素材。** Memory 应该帮助想象未来，而不是只保存过去。
6. **候选未来。** Candidate generator 应该生成可验证的未来预测，而不是浅层动作列表。
7. **LLM 深思。** LLM 应该在候选未来和关键权衡之间做战略判断，而不是只充当按钮选择器。
8. **预测误差学习。** 系统必须记录预测了什么、实际发生了什么、哪个模块应该改变。
9. **经过验证的执行。** LLM 不直接操作游戏；所有动作必须合法、可执行、基于当前状态。
10. **事实 / 记忆 / 派生知识分离。** 学到的信念不能污染事实层。
11. **Transition 记录。** 系统必须朝丰富的 preState / decision / postState / predictionError 记录前进。
12. **Checkpoint 安全。** 执行必须在有意义的不确定性后重新读取状态。
13. **稳定层保守晋升。** 稳定策略变化需要证据、条件、评估和回滚。
14. **实验层自由探索。** 激进探索可以存在，但必须与稳定策略隔离。
15. **可维护性。** 项目应该随着成长变得更清晰，而不是更混乱。

## 25. 什么算作进展

进展不是增加更多 prompt。
不是写更多文档。
不是堆更多记忆。
不是添加更多启发式规则。
不是让 LLM 说出更漂亮的复盘。

进展是认知脚手架闭环变强。

好的进展包括：

* 更稳定的 canonical state；
* 更准确的 mechanics；
* 更好的战略第一印象；
* 更好的显著性检测；
* 更相关的记忆激活；
* 更有条件的记忆；
* 更好的候选未来；
* 更强候选覆盖；
* 更准候选预测；
* 更好的 LLM 深思包；
* 更安全的执行；
* 更完整的 transition；
* 更有用的 replay；
* 更准确的 prediction error attribution；
* 更可靠的 learning update；
* 更少重复失败；
* 更清晰模块边界；
* 更多来自真实失败的测试。

坏的进展包括：

* 添加绕过架构的策略规则；
* 让本地规则悄悄替代 LLM 战略判断；
* 把 context compression 做成普通摘要；
* 把 memory 做成无条件经验堆；
* 把 candidate generation 做成合法动作枚举；
* 把 LLM 反思当成事实；
* 为了短 prompt 丢掉决定性信息；
* 让 prompt 变成全量历史倾倒；
* 混合事实、记忆和派生主张；
* 依赖未测试的外部字段；
* 把模糊推断当作标签；
* 修改底层机制但没有测试；
* 创造一个有时能赢、但无法解释、回放或改进的系统。

## 26. 冲突解决

当目标冲突时，优先顺序是：

1. **真实游戏动作的安全性和合法性。**
2. **状态正确性和数据真相。**
3. **可观测性和可回放性。**
4. **LLM 核心战略玩家地位。**
5. **战略第一印象和候选未来质量。**
6. **战略游玩质量。**
7. **速度和 token 效率。**
8. **学习和适应。**
9. **便利性和短期开发速度。**

一个无法审计的更快 agent，比一个可以改进的较慢 agent 更差。

一个会破坏数据真相的聪明策略，比一个记录不确定性的保守策略更差。

一个通过悄悄替代 LLM 来赢的脚手架，并没有完成项目目的，除非它被明确标记为单独 baseline、solver 或实验系统。

一个遗漏决定性特征的短 prompt，比一个提供正确战略上下文的较长 prompt 更差。

一条听起来聪明但没有条件的记忆，比没有记忆更差。

一个安全但浅层的候选集合，可能成为 LLM 战略水平的上限。

## 27. 给未来智能体的永久指令

未来编码智能体在做结构性修改前，必须先内化本文档，并检查当前项目状态。

不要假设旧 handoff、debug、planning 或 experiment 文档中的实现细节仍然当前有效。

每个重大变更都应该回答：

```text
这属于哪一层？
它是否保留 LLM 核心战略玩家地位？
它是否强化预测型认知脚手架？
它是否帮助 LLM 更好地看见、记起、想象、深思、执行或学习？
它是否改进战略第一印象？
它是否改进显著性判断？
它是否改进记忆激活而不制造迷信？
它是否改进候选未来而不绕过验证？
它是否让预测误差更容易记录和归因？
它是否保持事实 / 记忆 / 派生知识分离？
它是否改进 transition recording 和 replay？
它是否保持外部依赖在 adapter 后面？
它是否明确人类数据可靠性？
它是否包含必要测试或 fixture？
它是否可以评估？
它是否可以回滚？
```

如果答案不清楚，该变更应该在实现前重新设计。

---

## 28. 持久的一句话总结

构建一个真实游戏中的《Slay the Spire 2》LLM 智能体：LLM 保持核心战略玩家地位，而预测型认知脚手架把原始状态转化为战略第一印象，激活相关记忆，生成可验证的候选未来，让 LLM 在紧凑但战略完整的上下文中深思，并通过安全执行、transition 记录、回放评估和预测误差学习，使 LLM 在重复游玩中逐步接近它自身能够达到的最强战略表现。
